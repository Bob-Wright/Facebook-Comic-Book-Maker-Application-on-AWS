Facebook Comic Book Maker Application on AWS

Introduction.

This tutorial repository contains information and instructions about how to set up an example Facebook application on an Amazon Web Services (AWS) hosted website. The site runs on a Bitnami Ubuntu Linux, Apache, mySQL, and PHP (LAMP) stack server in the Amazon cloud. This document assumes that the LAMP stack is already set up and functional and that its command console is accessible through a terminal program like PuTTY, and it assumes some familiarity with the LAMP stack and its components. It is also assumed that the site already has an SSL certificate installed and operating correctly. The tutorial will initially focus more upon the server application setup than it will upon the details of the application's functions and code. All of the code for the application itself is also included in the repository, and after the filesystem setups this tutorial continues with explanations about the structure of the builder application and the functions of its programs.

Setting up the application on Facebook requires that the application code is already written and functional, and that the host site URLs are accessible and active. So we will have to set all of that up before we can add the application to Facebook itself. The tutorial concludes with a few screenshots that provide a brief overview of setting the application up on Facebook.

It is suggested that you read over the tutorial completely before you begin performing the operations described. Additionally some of the example code will require editing to suit your environment and domain before it can be used.

For this example application we will use the idea of a comic book maker or "builder" named Storybook that will create and host web comics from user provided input data. The entry point for the comic book builder is from a comics gallery page. Once the comic is created it will be hosted on the website and displayed in the gallery along with other comics. The gallery on our actual Storybook site contains a few comics besides this example.

The comics created by the builder use a responsive design and incorporate the Bootstrap framework, so the site responds to the viewing device characteristics. The builder itself is probably best viewed on a laptop or desktop.

The first action required in the comic book maker application is for the user to login or authenticate themselves to the application as a Facebook user. Only Facebook users can upload content and create comics, but anyone can view the comics.

Besides creating and hosting the comics, the application also has some other functions.

One function that appears is a "Like" button to encourage a user to give the application a "Like" on Facebook. In our example we have this button on the user login page, and if the user has already "Liked" the application the button is highlighted. Once the user has provided the content and created a comic, the application then gives the user an opportunity to download a stand-alone archive of their comic book, and it gives the user an opportunity to easily share their comic on Facebook. The application sets up the Open Graph image and text that Facebook displays if the comic is shared.

If a user logs back into the application after they have already created a comic, they will be given the opportunity to share their comic to Facebook, or they may choose to delete their content. Allowing a user to delete their content is a requirement for Facebook applications.

The first action a user will take after successfully logging in via their Facebook account is to answer a number of "fill in the blank" questions about the comic. For example, what is the comic's name? What is the author's name? Following this is a series of dialogs wherein the user will choose and upload such content as the page image files or the text caption files for the comic.

So the foregoing provides an overview of the application's functionality. Now let's look at the server setup and folder hierarchy to implement these functions.

Users and Folder hierarchy.

We want to set up a web site that will allow users to upload content that the site will incorporate into web pages for display. We want to do this in a way that does not require us to trust users. We need to control access to certain folders in such a way as to allow our code to write and read website user uploaded content with several security considerations.

First, we do not allow any online user to upload any content to our DOCUMENT_ROOT

Second, since we do want to display uploaded user content we need to set up a place to upload and store the content.

Third, we need to set up access permissions and ownership for those upload directories

If this is done we can enjoy a more secure website and still allow the upload and display of user uploaded content.

Note that apache2$ is used as a simplified command prompt in the following descriptions and examples.

Set up a sessions database for PHP.

We want to set up a sessions handler database store using our mySQL server for our PHP sessions data. The philosophy of why this might be of benefit is beyond our discussion, but there are various strong reasons that you might want to do this.

This requires a sessions handler function to interface the PHP with the sessions database. We use the Zebra_Session code which is described as:
"A drop-in replacement for PHP's default session handler which stores session data in a MySQL database, providing both better performance and better security and protection against session fixation and session hijacking"
The code can be found on the Zebra_Session author's page.

Setup the Three Application Folders

As we saw previously, there are three more folders used in our hierarchy, a ComicsUser folder, a Storybook folder, and a Comics folder. These folders can easily be populated by copying the files from the repository and editing as needed. A most important distinction exists between these three folders and the four we just discussed, and that revolves around what entity might access their contents. The first four folders are all accessed by PHP code on the server and are unseen by the browser. But parts of these next three folders appear as subfolders beneath the htdocs folder, and as such they are visible to the HTML code and the browser. Their htdocs folders appear as though they were actually subfolders within the DOCUMENT_ROOT htdocs folder. Recall that we edited the three sets of configuration files that configure this Apache "alias" feature earlier.

Our first folder, the Storybook folder, is structured as shown in the directory tree below. The programs in this folder accomplish the majority of the application's actual comic maker functions. The main entry here is through the OauthPortal.php program which uses the files in the ComicsUser folder described later to log a user into the application using their Facebook credentials. OauthPortal.php is also the callback entry once a user has been authenticated by Facebook, and if the user logon was successful it then invokes the index.php code in this folder.

The index.php program manages the data gathering functions of the builder by using subprograms to obtain each value needed to create the comic. Each subprogram checks the value inputted for that data item and may format it according to requirements. An example of this sort of subprogram that obtains a single value is cardTitle.php which has the sole function of obtaining that value as an input from the user. Another sort of subprogram invoked by the index.php code obtains files uploaded by the user as content for the comic. One example of this sort of subprogram is getComic.php which serves to obtain the images that will make up the comic pages. All of the subprograms that gather files as input (as opposed to user entered values) invoke the JavaScript program vpb_uploader.js. That code performs the file selection and sorting functions and parses various information values about each file it uploads and passes each one in turn to the imgUploader.php program where it is processed and stored and where the database manager is updated. In some cases the user content files may be text, not images, as in getComicCaptions.php for example, and in that case we still use vpb_uploader.js but instead it will invoke txtUploader.php to process each uploaded text file. The index.php code can invoke different subprograms depending upon the application setup, for example reference the "txt" flag files in the includes folder.

Once all of the content and values necessary to create a comic are gathered by each dialog in turn, we will run the Yield.php program which sets up and manages the creation of the comic using the gathered data, invokes Comic.php to actually construct the comic book code, invokes code that creates a ZIP archive of the comic which may be downloaded, creates an entry in the site comic books gallery, and allows the comic book to be "Shared" to Facebook. The comic that the builder creates uses only HTML, CSS, and JavaScript so it requires only a browser to run. It also uses responsive design including the Bootstrap framework to support a wide variety of devices. There are a few housekeeping and function execution programs included in this folder as well
